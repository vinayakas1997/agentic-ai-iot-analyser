# Issue 4 - Prompt Enhancement Suggestions (2026-07-07)

## Overview

This document identifies prompt enhancement opportunities for the Manager Agent component. The current prompts are production-ready and well-structured, but there are areas where additional guidance could improve robustness and consistency.

---

## Identified Issues

### 1. Missing Error Recovery Guidance for Ambiguous Inputs

**Location:** `extract_slots.md` (lines 45-102)

**Current State:**
- The prompt defines `intent_mode: "unclear"` as a possible value
- Examples show clear cases (single machine, joint analysis, etc.)
- No explicit guidance on what to do when input is genuinely ambiguous or conflicting

**Issue:**
When users provide conflicting or highly ambiguous input (e.g., "show me sales for the past week and last month" with two time references, or "what about the fruits and the Vinayaka line" with unclear scope), the prompt doesn't specify:
- Whether to default to a specific mode
- How to handle multiple competing interpretations
- When to flag ambiguity vs. making a best-effort extraction

**Recommendation:**
Add explicit rules for ambiguous/conflicting input handling:

```markdown
- When time references conflict (e.g., "last week" and "past 2 days"), keep both in time_start_raw and time_end_raw, set intent_mode to "unclear"
- When multiple machines are mentioned without clear joint vs. per-slot intent, default to "joint" mode and set intent_mode to "unclear"
- For highly ambiguous inputs where extraction is unreliable, set all slots to null except line_mentions, and set clarification.session_intent to "meta_question"
```

**Impact:**
- More predictable behavior on edge cases
- Better downstream handling of ambiguous requests
- Reduced need for post-processing ambiguity resolution

### 2. Missing Multi-turn Clarification Strategy

**Location:** `extract_slots.md` (lines 5-102)

**Current State:**
- Session state is preserved with `lookup_locked` slots
- Follow-up turns include all machines from session state
- The prompt says "Never remove or re-resolve a slot with lookup_locked=true unless the explicitly gives a new name"

**Issue:**
There's no explicit guidance on:
- When to ask clarifying questions vs. proceeding with assumptions
- How to handle partial corrections (e.g., "actually skip the first one")
- What to do when says "do the other one" without specifying which analysis
- How to interpret vague follow-ups like "tell me more" without a specific target

**Recommendation:**
Add clarification strategy rules:

```markdown
- On follow-up turns, if provides partial information (e.g., "actually skip AM307A"), only update the specific slot mentioned, preserve all others
- When says "the other one" or "do that one" without specifying, set clarification.active_mention to null and intent_mode to "unclear"
- For vague follow-ups like "tell me more" or "explain that", set session_intent to "advisory" and look for a target in the prior message
- If confirms with "yes" or "go" after proposals are shown, set aim_exploration.action to "confirm" with the currently displayed proposal index
```

**Impact:**
- More consistent multi-turn behavior
- Better handling of partial corrections
- Reduced frustration from misunderstood follow-ups

### 3. Missing Dataset Join Logic

**Location:** `propose_analysis_plans.md` (lines 14-58)

**Current State:**
- The prompt receives `join_catalog` as input
- Rules say "Do not invent tables, columns, or joins not in context"
- Examples show joins but don't explain the logic for suggesting them

**Issue:**
The prompt doesn't provide explicit guidance on:
- How to evaluate which joins are valuable vs. trivial
- What makes a join "interesting" from a business perspective
- How to handle cases where multiple valid joins exist
- Whether to suggest all possible joins or only the most useful ones

**Recommendation:**
Add join selection criteria:

```markdown
- Prioritize joins that enable cross-domain analysis (e.g., joining quality with production, or sales with inventory)
- Avoid joins on generic keys (e.g., batch_id alone) unless it enables meaningful aggregation
- When multiple valid joins exist, suggest the one that enables the most diverse set of analyses
- For each proposed join, explain the business value in what_you_might_see (e.g., "This join lets you correlate defect rates with supplier performance")
```

**Impact:**
- More meaningful analysis proposals
- Better justification for join suggestions
- Reduced proposal redundancy

### 4. Enhancement Opportunity: Few-Shot Examples for Edge Cases

**Location:** `extract_slots.md` (lines 103-444)

**Current State:**
- 20+ examples covering common cases
- Good coverage of tier-1, tier-2, advisory, and meta_question scenarios
- Examples are clear and well-structured

**Issue:**
Missing examples for:
- Partial corrections ("actually skip the first machine")
- Multi-line joint analysis with different aims per line
- Time range corrections ("no, I meant last month not last week")
- Dataset-specific queries on the same line ("what about the error table")
- Reusing saved analyses ("same as last time")

**Recommendation:**
Add 5-7 new examples covering:

1. **Partial correction:**
```
User: "forget AM307A, show me aims for fruits test"
Output: { clarification: { skip_mentions: ["AM307A"], active_mention: "fruits test", ... }, line_mentions: ["fruits test"], ... }
```

2. **Multi-line with per-slot aims:**
```
User: "cost on Vinayaka, sales on LINE_B"
Output: { scope: { intent_mode: "per_slot", ... }, line_slots_detail: [ { mention: "Vinayaka", aim_raw: "cost" }, { mention: "LINE_B", aim_raw: "sales" } ] }
```

3. **Time range correction:**
```
User: "no, I meant last month not last week"
Output: { clarification: { time_intent: { action: "set_phrase", phrase: "last month" } }, time_raw: "last month" }
```

4. **Dataset-specific query:**
```
User: "what about the error table"
Output: { clarification: { dataset_mentions: ["error"], active_line: "FRUITS_TEST" }, line_mentions: ["FRUITS_TEST"] }
```

5. **Reusing saved analysis:**
```
User: "same as last Vinayaka run"
Output: { clarification: { reuse_alias: "last Vinayaka run" } }
```

**Impact:**
- Better few-shot learning for the model
- More consistent handling of edge cases
- Reduced need for post-hoc corrections

### 5. Enhancement Opportunity: Session Goal Propagation Strength

**Location:** `propose_analysis_plans.md` (lines 5-58)

**Current State:**
- `session_goal` is passed to the prompt
- Rule line 50 says "When session_goal is set, align proposals with that goal"
- No guidance on how strongly to bias toward the goal

**Issue:**
The prompt doesn't specify:
- Whether to prioritize the session goal over_explore_intent
- How to handle cases where session_goal conflicts with_explore_intent
- Whether to explicitly mention the session goal in proposals
- How to diversify proposals when the goal is very narrow

**Recommendation:**
Add explicit goal propagation rules:

```markdown
- When both session_goal and_explore_intent are set, prioritize session_goal as the primary focus, use_explore_intent to diversify within that focus
- If session_goal is "cost optimization", all proposals should relate to cost, but_explore_intent can suggest different cost angles (efficiency, waste reduction, pricing)
- Explicitly reference the session_goal in at least one proposal's title or aims text
- When session_goal is narrow (e.g., "cost by supplier"), still return 3 proposals but vary the dimension (cost by supplier, cost by region, cost by time)
```

**Impact:**
- More coherent proposal sets aligned with intent
- Better balance between focus and diversity
- Clearer traceability from's stated goal to proposals
