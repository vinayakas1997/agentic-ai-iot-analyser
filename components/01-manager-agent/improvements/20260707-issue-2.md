# Database Schema Optimization Opportunities

## Overview

This document identifies potential database schema improvements for the Manager Agent component. The current schema is well-structured with no redundant tables, but there are optimization opportunities for scalability and consistency.

## Current Schema Structure

### Active Tables
- `users` - User authentication and management
- `events` - Message bus queue for event-driven architecture
- `results` - Analysis results storage
- `chat_history` - Chat conversation turns with UI/schema snapshots
- `global_registry` - Canonical IoT line catalog (single source of truth)
- `task_registry` - User-specific task definitions
- `manager_sessions` - Session state with optimistic concurrency control

## Identified Issues

### 1. Inconsistent ID Types

**Location:** `events` table (migration 001_initial.sql:13)

**Current:**
```sql
event_id UUID DEFAULT gen_random_uuid() UNIQUE NOT NULL
_id TEXT NOT NULL
```

**Issue:**
- `event_id` uses UUID (correct)
- `_id` uses TEXT (inconsistent with `users.id` which is UUID)
- This prevents foreign key constraints and type safety

**Recommendation:**
```sql
_id UUID NOT NULL REFERENCES users(id)
```

**Impact:**
- Enables referential integrity
- Prevents orphaned records
- Better type safety in queries

### 2. Chat History Snapshot Bloat

**Location:** `chat_history` table (migration 008_chat_history_snapshots.sql)

**Current:**
```sql
CREATE TABLE chat_history (
    id          SERIAL PRIMARY KEY,
   _id     TEXT NOT NULL,
    session_id  TEXT NOT NULL,
    role        TEXT NOT NULL,
    content     TEXT NOT NULL,
    turn_index  INTEGER,
    ui_snapshot JSONB,        -- Can be large
    schema_snapshot JSONB,    -- Can be large
    created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

**Issue:**
- `ui_snapshot` and `schema_snapshot` can grow very large (hundreds of KB each)
- These snapshots are only needed for specific UI states
- Bloats the main chat_history table, affecting query performance
- Makes table backups and maintenance slower

**Recommendation:**
Split into two tables:

```sql
-- Main chat history (lean)
CREATE TABLE chat_history_messages (
    id          SERIAL PRIMARY KEY,
   _id     TEXT NOT NULL,
    session_id  TEXT NOT NULL,
    role        TEXT NOT NULL,
    content     TEXT NOT NULL,
    line_name   TEXT,
    node        TEXT,
    turn_index  INTEGER,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Snapshot storage (separate, can be archived independently)
CREATE TABLE chat_history_snapshots (
    id              SERIAL PRIMARY KEY,
    message_id      INTEGER REFERENCES chat_history_messages(id),
    ui_snapshot     JSONB,
    schema_snapshot JSONB,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

**Benefits:**
- Faster chat history queries (no large JSONB columns)
- Independent snapshot archival/cleanup
- Better indexing on core message fields
- Reduced table bloat

### 3. Global Registry Synonyms Performance

**Location:** `global_registry` table (migration 003_manager_tables_fresh.sql:10)

**Current:**
```sql
synonyms JSONB,
-- Index exists:
CREATE INDEX idx_global_registry_synonyms ON global_registry USING GIN (synonyms);
```

**Issue:**
- GIN index on JSONB array works but has limitations:
  - Complex synonym queries require JSON operators
  - Performance degrades with large synonym arrays
  - Cannot easily add synonyms without updating entire row

**Recommendation:**
Create a separate `registry_synonyms` table:

```sql
CREATE TABLE registry_synonyms (
    id              SERIAL PRIMARY KEY,
    line_name       TEXT NOT NULL,
    dataset_name    TEXT NOT NULL,
    synonym         TEXT NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (line_name, dataset_name, synonym)
);

CREATE INDEX idx_registry_synonyms_line ON registry_synonyms(line_name, dataset_name);
CREATE INDEX idx_registry_synonyms_synonym ON registry_synonyms(synonym);
```

**Benefits:**
- Better query performance for synonym lookups
- Easier synonym management (add/remove without row lock)
- Cleaner data model
- Can archive old synonyms independently

### 4. Results Table Referential Integrity

**Location:** `results` table (migration 001_initial.sql:30-39)

**Current:**
```sql
event_id UUID,  -- Nullable, no foreign key
```

**Issue:**
- `event_id` is nullable with no constraint
- No validation that referenced event exists
- Can have orphaned result records

**Recommendation:**
```sql
event_id UUID REFERENCES events(event_id)
```

**Impact:**
- Ensures data integrity
- Prevents orphaned results
- Better query reliability

## Migration Strategy

### Phase 1: Low-Risk Changes
1. Add foreign key to `events.user_id` (requires data migration)
2. Add foreign key to `results.event_id`

### Phase 2: Structural Changes
1. Split `chat_history` into messages + snapshots
2. Migrate existing data to new structure
3. Update application code to use new tables

### Phase 3: Optimization
1. Create `registry_synonyms` table
2. Migrate synonyms from JSONB array
3. Remove `synonyms` column from `global_registry`
