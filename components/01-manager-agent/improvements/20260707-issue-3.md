# Issue 3 — UI Fix & Enhancement: Manager Agent Surface (2026-07-07)

## Issues Identified

### 1. Same Manager decision rendered in three places, in three different formats
"Line: FRUITS_TEST" + "Aims: ..." appears in the Chat bubble (as freeform markdown "Plan" block), again in Outputs under "Active plan" / "Ready for planner," and the resolved line also sits in its own Context card. One Manager decision, three renderings, no single source of truth on screen. If a user wants to know "what did the Manager actually lock in," they have to cross-check three panels that could theoretically drift out of sync.

### 2. "Line match" and "Line" are redundant cards showing the same resolved fact twice
"Line match: Vinayaka → FRUITS_TEST (synonym)" already tells you the resolution *and* the reasoning (synonym match). The separate "Line" card just repeats the output with none of the reasoning. The reasoning card is the valuable one — the second card is dead weight.

### 3. "Suggested aims" (sidebar) never visibly connects to the proposals it actually produced
Sidebar says "average cost by fruit / sales by region" as suggested aims, but the actual proposals in Chat/Outputs are about cost+quantity, cost+revenue, cost+discount — none of which is "sales by region." Either the sidebar is showing stale/generic suggestions unrelated to this session, or there's no traceability from "suggested aim" → "which proposal fulfilled it." For a Manager agent whose whole value is turning ambiguous asks into scoped aims, that link should be visible, not implicit.

### 4. No correction affordance on Manager's interpretation before handoff
Line match, dataset scope, "No time filter" are all presented read-only. If the Manager mis-resolves ("Vinayaka" → wrong line, or wrong dataset in scope), the user's only lever is typing more chat text and hoping the Manager re-parses it — there's no click-to-edit on the resolved slots. Given Manager = interpretation/scoping layer, that's arguably its most important failure mode to design for, and right now the UI has zero affordance for it.

### 5. Handoff payload narrower than what the Manager actually decided
"READY FOR PLANNER" showed only Line + Aims. But the proposal the user picked also carried Datasets, Join, and (implicitly) which columns/filters apply. If those don't travel forward in the visible handoff, either they're silently passed via session state (fine, but then don't show partial info — show all of it or none), or they're genuinely dropped.

### 6. Context chips only appear on click (no per-turn context visibility)
The context bar at the top of ChatSection only updated when the user clicked a specific turn. This meant the user had to actively interact to see "what the Manager knew" at each step. Each Q&A pair should carry its own context inline so the chat history is self-documenting.

### 7. No pipeline progress visibility in Outputs
After the Manager confirms a plan and hands off to Planner/Executor, there was no visual indication of which stage the pipeline was in. The user couldn't tell at a glance whether the Planner was working, the Executor was running, or everything was complete. The existing ExecutionProgress component showed detailed logs but no high-level pipeline status.
