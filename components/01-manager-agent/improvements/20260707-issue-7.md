# Issue 7 — Outputs Panel Redesign: Results-Only (2026-07-07)

## Overview

The Outputs panel currently mirrors content that now belongs in Chat (Manager Decision, Proposals, Suggested aims), while providing no useful state for the gap before Planner/Execution produce results. When results do arrive, they are rendered as raw JSON with no structure, stage identity, or visual hierarchy.

---

## Identified Issues

### 1. Outputs Mirrors Manager Content Now Owned by Chat

**Location:** `OutputSection.tsx` — Proposals, ManagerDecisionCard, Suggested aims, Saved plans, ActionButtons, Still needed

**Current State:**
- `ManagerDecisionCard` (summary variant) rendered inline
- Proposals card with aims/datasets/join/you-might-see
- Suggested aims as clickable buttons
- Saved plans list
- "Still needed" list
- ActionButtons (Confirm 1/2/3, Go, etc.)

**Issue:**
Chat now owns all Manager decision rendering (structured plan view, selectable proposal cards, quick-reply action chips). Outputs duplicating this content creates two problems:
- Users see the same information twice
- Outputs loses its identity as the "results" panel

---

### 2. No Honest Empty/Waiting State

**Location:** `OutputSection.tsx` — ExecutionProgress

**Current State:**
- `ExecutionProgress` returns `null` when `executionEvents` is empty
- Outputs panel shows Session complete card + nothing else
- No visual indication that the system is waiting on Planner

**Issue:**
When Manager completes but Planner hasn't started yet, the panel is effectively empty. This is a real design moment — the user needs to know "what's next" — but it's glossed over with an empty gray box.

---

### 3. Results Render as Raw JSON With No Structure

**Location:** `OutputSection.tsx` — ExecutionProgress result rendering

**Current State:**
- `task.complete` events show results as `<pre>` JSON inside `<details>` elements
- No stage identity (green for Execution, yellow for Planner)
- No structured layout (KPIs, tables, charts)
- No persistent "which stage produced this" tag

**Issue:**
When execution results do arrive, they should look like a dashboard (KPI strip, data table, insight note) — not a debugging console. Each result block should carry a stage-colored tag identifying its provenance.

---

### 4. No Persistent Stage-Provenance Tag

**Location:** `OutputSection.tsx` — All result rendering

**Current State:**
- No result blocks have stage badges or labels
- No visual distinction between Planner output and Execution output

**Issue:**
Once Planner starts contributing (query plan previews, intermediate results) alongside Execution, this panel will hold output from multiple stages. Each block needs a `result-badge` + `result-label` to identify its source.
